# RAG-система

## 1. Назначение

Система принимает вопрос пользователя на естественном языке и возвращает два набора текстовых фрагментов (вырезок) из заданного исходного документа (`DM2024_module9.txt`). Фрагменты делятся на «короткие» (размером около 15 токенов) и «длинные» (около 100 токенов). Поиск основан на косинусном расстоянии между векторными представлениями (эмбеддингами) вопроса и предварительно индексированных фрагментов. Генерация ответа с помощью большой языковой модели (LLM) не производится.

## 2. Общая архитектура

Система состоит из четырёх сервисов, оркестрируемых через `docker-compose.yml`:

- **PostgreSQL** с расширением `pgvector` – хранит эмбеддинги фрагментов.
- **Bootstrapper** – однократно (или при перестроении) разбивает исходный текст на фрагменты, вычисляет эмбеддинги и заполняет базу данных.
- **Backend** – FastAPI-приложение, принимающее вопросы, вычисляющее эмбеддинг вопроса и выполняющее поиск.
- **Frontend** – статический HTML/CSS/JS интерфейс для отправки вопросов и отображения ответов.
- **Nginx** – выступает в роли reverse-proxy, обеспечивает маршрутизацию `/api/` на backend, статики – на frontend, а также добавляет самоподписанный TLS (порт 8443) и HTTP-редирект.

Все сервисы работают в одной сети `ragnet`.

## 3. Компоненты и их взаимодействие

### 3.1 PostgreSQL (pgvector)

- Образ: `pgvector/pgvector:pg15`.
- Инициализация: скрипт `postgresql/init.sql` создаёт расширение `vector` и две таблицы:
  - `documents_short` – для коротких фрагментов.
  - `documents_long` – для длинных фрагментов.
- Каждая таблица имеет поля:
  - `id SERIAL PRIMARY KEY`
  - `content TEXT` – текст фрагмента.
  - `embedding vector(768)` – эмбеддинг размерности 768 (соответствует модели `intfloat/multilingual-e5-base`).
- Для ускорения поиска созданы индексы `IVFFlat` с метрикой `vector_cosine_ops` (косинусное расстояние).
- Доступ к БД через переменную окружения `DATABASE_URL`.

### 3.2 Bootstrapper

Сервис, выполняющий однократную загрузку и векторизацию исходного текста. Запускается отдельной службой в `docker-compose.yml` с переопределённой командой. Логика реализована в `backend/app/bootstrapper.py`.

**Алгоритм:**

1. Проверяет наличие записей в таблице `documents_short`. Если записи есть и переменная окружения `REBUILD_EMBEDDINGS` не равна `true`, то завершает работу (данные уже загружены).
2. При необходимости очищает таблицы (`TRUNCATE`).
3. Читает все `.txt` файлы из каталога, указанного в `PDF_PATH` (в системе – `/app/sources`).
4. Для каждого файла:
   - Разбивает текст на **короткие чанки**: размер 15 токенов, перекрытие 3 токена.
   - Разбивает текст на **длинные чанки**: размер 100 токенов, перекрытие 20 токенов.
   - Токенизация выполняется с помощью токенизатора модели `intfloat/multilingual-e5-base`.
5. Накопленные чанки обоих типов преобразуются в эмбеддинги:
   - Каждому фрагменту добавляется префикс `"passage: "` (требование модели e5).
   - Вызов `model.encode()` с `normalize_embeddings=True` (нормировка для косинусного расстояния).
6. Эмбеддинги сохраняются в соответствующие таблицы БД.

**Переменные окружения для bootstrapper:**
- `DATABASE_URL` – подключение к PostgreSQL.
- `REBUILD_EMBEDDINGS` – если `true`, то очищает таблицы и перестраивает индексы независимо от наличия данных.
- `PDF_PATH` – путь к каталогу с исходными `.txt` файлами.

### 3.3 Backend (FastAPI)

Сервис на Python с использованием `fastapi`, `sentence-transformers`, `psycopg2` и `pgvector`.

**Запуск:** через `uvicorn app.main:app --host 0.0.0.0 --port 8000` (в Dockerfile – CMD).

**Модель:** загружается один раз при старте приложения – `intfloat/multilingual-e5-base`. Кэш модели сохраняется в томе `backend_cache:/models/cache`, чтобы избежать повторной загрузки при перезапуске.

**Эндпоинт:** `POST /api/question`

- **Тело запроса (JSON):** `{"question": "текст вопроса"}` – валидируется через `UserIn` (минимальная длина 1, максимальная 1000 символов).
- **Ответ (JSON):** `{"short": ["фрагмент1", ...], "long": ["фрагмент1", ...]}` – максимум по 5 фрагментов каждого типа (константы `TOP_K_SHORT=5`, `TOP_K_LONG=5`).

**Алгоритм обработки запроса:**

1. К вопросу добавляется префикс `"query: "` (требование модели e5).
2. Вычисляется эмбеддинг вопроса через `model.encode([text], normalize_embeddings=True)`. Результат – список из 768 чисел с плавающей точкой.
3. Выполняются два SQL-запроса к таблицам `documents_short` и `documents_long`:
   ```sql
   SELECT content, embedding <=> %s::vector AS distance
   FROM documents_short
   ORDER BY embedding <=> %s::vector
   LIMIT %s
   ```
   Оператор `<=>` вычисляет косинусное расстояние (1 - косинусное сходство). Чем меньше расстояние, тем ближе фрагмент к вопросу.
4. Из результатов извлекаются только тексты `content` и возвращаются в ответе.

**Важно:** backend не выполняет никакой генерации текста. Возвращаются именно исходные фрагменты из учебника.

### 3.4 Frontend

Статический веб-интерфейс, собранный в образ nginx. Исходные файлы: `frontend/index.html`, `frontend/nginx.conf`.

**Функции:**
- Текстовое поле для ввода вопроса (счётчик символов, ограничение 1000).
- Кнопка отправки.
- При отправке – POST-запрос на `/api/question` (проксируется на backend через nginx-reverse-proxy).
- Отображение ответа: сначала выводятся короткие фрагменты (секция «Кратко»), затем длинные («Подробнее»). Если фрагментов нет – выводится «Нет ответа».
- Обработка ошибок и индикатор загрузки.

Статика обслуживается nginx-контейнером `frontend` на порту 80. Доступ к нему через основной nginx-прокси.

### 3.5 Nginx reverse-proxy

Основной вход в систему. Собирается из `nginx/Dockerfile`, использует `nginx/nginx.conf` и `nginx/entrypoint.sh`.

**Конфигурация:**
- Порт 80: редирект на HTTPS (301).
- Порт 443: SSL с самоподписанным сертификатом (генерируется в `entrypoint.sh` при отсутствии).
- Маршруты:
  - `/api/` – прокси на `backend:8000` с передачей заголовков.
  - `/` – прокси на `frontend:80` (статический интерфейс).

**TLS:** сертификат генерируется на лету с помощью `openssl req -x509` для `CN=localhost`. В реальной эксплуатации требуется заменить на валидные сертификаты.

## 4. Переменные окружения (файл .env)

- `REBUILD_EMBEDDINGS` – управляет перестроением индексов в bootstrapper.
- `PDF_PATH` – внутри контейнера bootstrapper, указывает на каталог с исходными текстами (монтируется `./sources:/app/sources`).
- `DATABASE_URL` – используется и backend, и bootstrapper.
- `POSTGRES_*` – параметры для сервиса postgres.

## 5. Запуск и развёртывание

### Требования
- Docker и Docker Compose (версия не ниже 2.0).
- Исходный файл учебника должен лежать в каталоге `sources/DM2024_module9.txt` (или другой `.txt`, система возьмёт все файлы в этом каталоге).

### Шаги

1. Клонировать репозиторий проекта.
2. Убедиться, что в каталоге `sources/` присутствует хотя бы один `.txt` файл.
3. При необходимости изменить переменные в `.env` (например, пароль БД).
4. Выполнить:
   ```bash
   docker-compose up --build
   ```
5. Дождаться, пока bootstrapper завершит загрузку данных (в логах появится `BOOTSTRAPPER DONE`).
6. Открыть в браузере: `https://localhost:8443` (принять самоподписанный сертификат). HTTP-порт 8080 также доступен, но будет редиректить на HTTPS.

#### Примечание
Если возникают проблемы с nginx, то перейдите в директорию nginx и в терминале выполните:
```
sed -i 's/\r$//' entrypoint.sh
```

### Примечания по запуску

- Bootstrapper запускается параллельно с backend. Если backend начнёт принимать запросы раньше, чем bootstrapper заполнит БД, ответы будут содержать пустые списки. Для production рекомендуется добавить `depends_on` с условием готовности bootstrapper или использовать `healthcheck` в bootstrapper (не реализовано). В текущей версии bootstrapper запускается однократно и завершается; для перезапуска используйте `docker-compose up --force-recreate bootstrapper` или установите `REBUILD_EMBEDDINGS=true`.
- Модель `intfloat/multilingual-e5-base` весит около 1.1 ГБ и будет загружена при первом запуске backend и bootstrapper. Кэш сохраняется в томе `backend_cache`, поэтому повторные запуски быстрее.


## 6. Ограничения и возможные улучшения

- **Отсутствие LLM:** система не умеет переформулировать ответ или объединять несколько фрагментов в связный текст. Это сделано намеренно в соответствии с постановкой задачи (RAG без LLM).
- **Качество эмбеддингов:** модель `multilingual-e5-base` оптимизирована для поиска с префиксами `query:` и `passage:`. Другие модели могут дать иной результат.
- **Размер чанков:** параметры (15/100 токенов) подобраны эмпирически для данного учебника. Для других источников может потребоваться настройка в `bootstrapper.py` (константы `chunk_size`, `overlap`).
- **Индексы IVFFlat:** требуют обучения на данных. В `init.sql` индексы создаются без параметра `lists`, используется значение по умолчанию (обычно 1000 для больших таблиц). Для точной настройки следует добавить `WITH (lists = ...)`.
- **Безопасность:** самоподписанный TLS не подходит для production. Необходимо заменить сертификаты на выданные доверенным центром.
- **Масштабирование:** bootstrapper и backend используют одну и ту же модель, что приводит к дублированию памяти. Для уменьшения потребления можно вынести эмбеддинг в отдельный сервис.
